# OCR 기능 구현 방침

## 1. 현재 상태

현재 웹앱에는 건강검진 결과지 PDF/이미지를 업로드하는 OCR 데모 흐름이 구현되어 있습니다.

현재 구현:

- 파일 업로드 UI 제공
- `/risk/ocr/demo` API 제공
- `/risk/ocr/extract` API 제공
- 실제 OCR 엔진 연결 전까지는 데모 입력값을 반환
- 사용자가 OCR 기능의 사용 흐름을 이해할 수 있도록 화면상 스캔 애니메이션 제공

## 2. 제출 전략

이번 제출에서는 OCR을 핵심 기능이 아니라 "다음 단계 확장 기능"으로 두는 것이 안전합니다.

이유:

- 건강검진 결과지는 기관·연도·양식에 따라 레이아웃이 다름
- OCR 정확도는 이미지 품질, 회전, 흐림, 표 구조 인식에 크게 영향을 받음
- 잘못 추출된 수치가 위험예측에 들어가면 결과 신뢰도가 낮아질 수 있음
- 제출 마감이 촉박한 상황에서는 수동 입력 + 공공데이터 샘플 + OCR 데모가 더 안정적임

따라서 제출서류에는 다음과 같이 표현합니다.

> MVP에서는 건강검진 수치 수동 입력과 공공데이터 샘플 입력을 우선 구현하였다. OCR 업로드는 사용자의 입력 부담을 줄이기 위한 다음 단계 기능으로 설계했으며, 현재는 업로드 흐름과 API 확장 지점을 구현해 실제 OCR 엔진 연결이 가능하도록 구성하였다.

## 3. 실제 OCR 구현 후보

### 후보 1: Tesseract OCR

장점:

- 오픈소스
- 서버 내부에서 처리 가능
- 별도 과금 없음

단점:

- 한국어·숫자·표 구조 인식 품질 튜닝 필요
- Render 환경에 시스템 패키지 설치가 필요할 수 있음
- 건강검진표처럼 표가 많은 문서에는 후처리가 중요함

적합도:

- 로컬 데모에는 가능
- Render 무료 배포 환경에서는 세팅 부담 있음

### 후보 2: Google Cloud Vision OCR

장점:

- 문서 OCR 품질 우수
- 숫자 인식 안정적
- 이미지/PDF 처리 확장 가능

단점:

- 클라우드 계정 및 키 관리 필요
- 과금 가능성 있음
- 공모전 제출용 MVP에는 설정 부담이 있음

적합도:

- 실제 서비스화 단계에 적합

### 후보 3: Naver CLOVA OCR

장점:

- 한국어 문서 OCR에 강점
- 국내 서비스와 친화적
- 템플릿 기반 OCR 구성 가능

단점:

- API 키와 외부 서비스 설정 필요
- 양식별 템플릿 관리가 필요할 수 있음

적합도:

- 건강검진 결과지 양식이 어느 정도 정리되면 적합

### 후보 4: OpenAI Vision

장점:

- 표와 문맥을 함께 해석 가능
- "수축기혈압", "공복혈당" 같은 항목 의미 추출에 유리
- 결과를 JSON 형태로 구조화하도록 지시 가능

단점:

- 의료 정보 이미지 처리에 대한 개인정보·보안 고려 필요
- API 비용 발생 가능
- 결과 검증 로직이 반드시 필요

적합도:

- 프로토타입 고도화에는 유용
- 실제 서비스에서는 개인정보 처리방침과 동의 절차가 필요

## 4. 권장 구현 방향

이번 공모전 MVP 기준 권장안:

1. 현재는 OCR 데모 흐름 유지
2. 제출서류에는 "OCR 자동 입력은 다음 단계 기능"으로 명시
3. 실제 구현 시에는 Naver CLOVA OCR 또는 OpenAI Vision을 우선 검토
4. OCR 결과는 바로 예측에 넣지 않고 사용자 확인 단계를 거침
5. 잘못 추출된 값은 사용자가 직접 수정할 수 있게 함

## 5. OCR 처리 흐름 설계

향후 실제 OCR 구현 흐름:

1. 사용자가 건강검진 결과지 이미지 또는 PDF 업로드
2. 서버가 OCR 엔진에 파일 전달
3. OCR 텍스트 추출
4. 항목명 매핑
   - 수축기혈압
   - 이완기혈압
   - 공복혈당
   - 총콜레스테롤
   - HDL
   - LDL
   - 중성지방
   - 신장
   - 체중
   - 허리둘레
5. 숫자 단위 정규화
6. 비정상 범위 값 검증
7. 사용자 확인 화면 표시
8. 확인된 값만 `/risk/predict`로 전달

## 6. OCR 결과 검증 규칙

OCR 값은 다음 범위를 벗어나면 자동 반영하지 않고 사용자 확인이 필요합니다.

- 수축기혈압: 70~240
- 이완기혈압: 40~160
- 공복혈당: 50~400
- 총콜레스테롤: 80~400
- HDL: 10~150
- LDL: 30~300
- 중성지방: 30~800
- 키: 120~230
- 체중: 30~200
- 허리둘레: 40~160

## 7. 개인정보 처리 원칙

- 업로드 파일은 서버 파일시스템에 영구 저장하지 않음
- OCR 처리가 끝나면 원본 파일 폐기
- 사용자가 확인한 수치만 위험예측에 사용
- 외부 OCR API 사용 시 사용자 동의 필요
- 민감정보 처리 방침과 보관 기간 명시 필요

## 8. 제출 시 표현 문장

> OCR 기능은 건강검진 수치 입력 부담을 줄이기 위한 확장 기능이다. 현재 MVP는 업로드 UI와 API 확장 지점을 구현했으며, 실제 서비스 단계에서는 OCR 결과를 즉시 예측에 사용하지 않고 사용자 확인·수정 단계를 거친 뒤 위험예측에 반영한다. 업로드된 원본 파일은 서버에 영구 저장하지 않는 구조로 설계한다.

## 9. 남은 OCR 작업

- OCR 엔진 선택
- 샘플 검진 결과지 이미지 수집
- 항목명 매핑 사전 작성
- 표 구조 추출 테스트
- OCR 결과 검증 로직 구현
- 사용자 확인 화면 구현
- 개인정보 처리방침 반영
